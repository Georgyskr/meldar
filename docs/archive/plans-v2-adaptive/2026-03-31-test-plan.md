## Conventions

### File naming
- Unit tests: co-located with source â€” `src/server/discovery/parsers/__tests__/chatgpt.test.ts`
- API route tests: `src/app/api/discovery/<route>/__tests__/route.test.ts`
- Integration tests (real Anthropic API): `test/integration/<feature>.test.ts`
- Security tests: `test/security/<area>.test.ts`
- E2E tests: `test/e2e/<feature>.spec.ts` (Playwright)
- Performance benchmarks: `test/performance/<area>.bench.ts`

### describe/it patterns
```ts
describe('functionName', () => {
  describe('happy path', () => {
    it('does X when given Y', async () => { ... })
  })
  describe('error cases', () => {
    it('returns error when Z', async () => { ... })
  })
})
```

### Assertion style
- `expect(res.status).toBe(400)` â€” always assert HTTP status explicitly
- `expect(json.error.code).toBe('VALIDATION_ERROR')` â€” always assert structured error codes, not just status
- `expect(json.success).toBe(true)` â€” for success paths
- Never assert on exact AI response content in unit tests; assert on shape/schema compliance
- Never use `as any` â€” use proper types on all mocks

### [REVISED] Mock strategy for AI calls

Use this exact factory in every test file that touches Anthropic. Declare it at the top of the file before any imports, using `vi.hoisted`:

```ts
const { mockMessagesCreate } = vi.hoisted(() => ({
  mockMessagesCreate: vi.fn<Parameters<Anthropic['messages']['create']>>(),
}))

vi.mock('@anthropic-ai/sdk', () => ({
  default: vi.fn().mockImplementation(() => ({
    messages: { create: mockMessagesCreate },
  })),
}))
```

Stub a valid `tool_use` response like this:

```ts
mockMessagesCreate.mockResolvedValue({
  content: [
    {
      type: 'tool_use' as const,
      id: 'toolu_01',
      name: 'extract_screen_time',
      input: {
        apps: [{ name: 'Instagram', usageMinutes: 90, category: 'social' }],
        totalScreenTimeMinutes: 360,
        platform: 'ios',
        confidence: 'high',
      },
    },
  ],
})
```

For error paths: always use `mockRejectedValue(new Error('...'))` when the production code throws, NOT `mockResolvedValue({ error: '...' })`. The parsers (`parseChatGptExport`, `parseGoogleTakeout`) throw â€” they do not return error objects. The upload route's outer `catch` block pattern-matches thrown error messages to decide the HTTP status code (422 vs 500). Tests that simulate parse errors MUST use:

```ts
mockParseChatGptExport.mockRejectedValue(new Error('No conversations.json found in ChatGPT export'))
// â†’ upload route outer catch â†’ message.includes('No conversations.json') â†’ 422
```

NOT:
```ts
mockParseChatGptExport.mockResolvedValue({ error: 'No conversations.json' }) // WRONG â€” route never sees this
```

### [REVISED] Rate limiting mock â€” per-route specifics

Always mock the whole module. Include ALL limiters the module exports so no real limiter leaks in:

```ts
vi.mock('@/server/lib/rate-limit', () => ({
  checkRateLimit: vi.fn<typeof checkRateLimit>().mockResolvedValue({ success: true }),
  screentimeLimit: null,
  quizLimit: null,
  analyzeLimit: null,
  adaptiveLimit: null,
  checkoutLimit: null,
  loginLimit: null,
  resetLimit: null,
  subscribeLimit: null,
}))
```

Each route imports only its own limiter â€” but the mock must export all of them as `null` to prevent real Redis connections in any test that transitively imports `rate-limit.ts`. If a route imports a limiter that is NOT in the mock, Vitest exports `undefined`, the real Upstash client is not instantiated (because `getRedis()` returns null in test environments), but it is still safer to be explicit.

```ts
import JSZip from 'jszip'

async function makeBombZip(): Promise<File> {
  const zip = new JSZip()
  zip.file('big.json', '{}')
  const archive = await zip.generateAsync({ type: 'arraybuffer' })
  const loaded = await JSZip.loadAsync(archive)
  const entry = loaded.files['big.json'] as unknown as { _data: { uncompressedSize: number } }
  entry._data = { uncompressedSize: 600 * 1024 * 1024 } // 600 MB > 500 MB limit
  const blob = new Blob([await loaded.generateAsync({ type: 'arraybuffer' })])
  return new File([blob], 'export.zip', { type: 'application/zip' })
}
```

## 9. Known Gaps (Documented, Not Planned)

### Concurrent uploads to the same session

The `sourcesProvided` idempotency check reads the array before the atomic `array_append` write. Two concurrent uploads of the same platform from the same client would both pass the check, both extract, and both append â€” wasting AI credits. The DB `array_append` is atomic so there is no data corruption. No unit test covers this race condition; it would require a harness that issues two simultaneous HTTP requests. Add a `// known gap: concurrent upload TOCTOU` comment in the upload route test file.

### Cross-tab atom sync

`atomWithStorage` does not handle `storage` events from other browser tabs by default. Multi-tab behavior is intentionally untested.

### DataUploadHub component RTL tests

`DataUploadHub.tsx` has substantial UI logic (OCR/Vision branching, error-to-done revert on partial success, billing redirect). RTL unit tests are out of scope for this plan. The E2E test in section 5 provides smoke coverage. If dedicated component tests are added later: `src/features/discovery-flow/ui/__tests__/DataUploadHub.test.tsx` with `@testing-library/react`, mocking `fetch`, `waitForOcr`, and `extractText`.

### Network timeout in DataUploadHub.handleFile

The `fetch` call has no timeout. The catch block runs on network failure, setting `status: 'error'` or reverting to `'done'` if prior uploads existed. Partially covered by E2E tests only.

## 10. Coding Standards Enforcement

All test files must pass `pnpm biome check .` before merge.

- **No `as any`:** Use `vi.fn<Parameters<T>>()` with explicit type parameters.
- **`vi.hoisted` always first:** Declare mock factories at the top of the file inside `vi.hoisted(() => ({ ... }))`, before any imports that reference the mocked modules.
- **`describe('functionOrRouteName')`** for top-level groups. Nested `describe` for sub-scenarios.
- **`it('does X when Y')`** â€” active voice, describes observable outcome given the condition.
- **`NextRequest` typing:** Use `new Request(...) as unknown as NextRequest`. Match the pattern in `login.test.ts`. Never cast to `NextRequest` directly.
- **`beforeEach`/`afterEach` cleanup:** `vi.clearAllMocks()` in `afterEach`. `vi.unstubAllEnvs()` after any `vi.stubEnv` call. `localStorage.clear()` in `beforeEach` for atom tests.
- **No `console.log` in tests.** Use `expect` assertions.
- **Import order:** Biome enforces automatic sorting. Do not manually sort.
