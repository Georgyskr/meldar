## 1. Deployment Architecture

### Decision: Vercel Hobby (PoC/CV) -> Vercel Pro (MVP)

Single Next.js 16 application on Vercel. No separate backend service.

```
User -> Vercel Edge Network -> Next.js App (App Router)
                                 |-- Static pages (SSG/ISR)
                                 |-- /api/subscribe          -> Resend API
                                 |-- /api/upload/screentime  -> Anthropic API (Haiku 4.5)
                                 |-- /api/billing/checkout   -> Stripe Checkout
                                 |-- /api/billing/webhook    -> Stripe webhook handler
                                 |-- /api/quiz/submit        -> Neon Postgres
                                 '-- Database queries        -> Neon Postgres (Frankfurt)
```

**Vercel Hobby limits (free):**

| Resource | Limit | Sufficient until |
|----------|-------|------------------|
| Bandwidth | 100 GB/month | ~50K page views/month |
| Serverless execution | 100 GB-hours/month | ~200K function invocations |
| Serverless timeout | 60 seconds | All PoC/CV endpoints |
| Build minutes | 6,000/month | ~200 builds/month |
| Preview deployments | Unlimited | All PRs |
| Edge Middleware | Included | Rate limiting |

**When to upgrade to Vercel Pro ($20/month):**
- Team collaboration needed (Pro supports multiple team members)
- Log retention needed beyond 1 hour
- Serverless function size > 250 MB (unlikely without `sharp`)
- Need for Vercel Cron with < 1-day intervals

## 2. Database

### Decision: Neon Postgres (Frankfurt) + Drizzle ORM. Mandatory from PoC Week 1.

**Why mandatory in PoC (not deferred to MVP):**
1. Stripe webhook `checkout.session.completed` needs server-side persistence for payment verification
2. Shareable X-Ray URLs (`/xray/[id]`) need server-side data storage
3. Founding 50 counter needs a real count (`SELECT COUNT(*) FROM payments`)
4. AI cost ceiling tracking needs a fallback if Redis is down (secondary concern)

**Neon free tier:**

| Resource | Limit |
|----------|-------|
| Storage | 512 MB |
| Compute | 0.25 vCPU, auto-suspends after 5 min idle |
| Branches | Unlimited |
| Region | Frankfurt (eu-central-1) -- GDPR compliant |
| Point-in-time recovery | 7 days |

**Dependencies to add:**

```bash
pnpm add @neondatabase/serverless drizzle-orm
pnpm add -D drizzle-kit
```

**PoC schema (3 tables):**

```
src/server/db/
  schema.ts       # Drizzle schema definitions
  client.ts       # Neon serverless connection
  migrations/     # Generated by drizzle-kit
```

```typescript
// src/server/db/schema.ts
import { pgTable, uuid, text, integer, timestamp, jsonb, boolean } from 'drizzle-orm/pg-core';

// Table 1: Subscribers (existing /api/subscribe migrated from fire-and-forget to persisted)
export const subscribers = pgTable('subscribers', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull().unique(),
  source: text('source').notNull().default('landing'),  // 'landing' | 'quiz' | 'xray'
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
});

// Table 2: Payments (Stripe webhook data)
export const payments = pgTable('payments', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull(),
  stripeCustomerId: text('stripe_customer_id').notNull(),
  stripeSessionId: text('stripe_session_id').notNull().unique(), // idempotency key
  product: text('product').notNull(),           // 'time_audit' | 'app_build'
  amountCents: integer('amount_cents').notNull(),
  currency: text('currency').notNull().default('eur'),
  status: text('status').notNull().default('completed'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
});

// Table 3: X-Ray results (shareable via /xray/[id])
export const xrayResults = pgTable('xray_results', {
  id: uuid('id').primaryKey().defaultRandom(),
  sessionId: text('session_id').notNull(),      // anonymous session cookie
  apps: jsonb('apps').notNull(),                // [{name, usageMinutes, category}]
  totalMinutes: integer('total_minutes').notNull(),
  topApp: text('top_app').notNull(),
  insight: text('insight').notNull(),
  isPublic: boolean('is_public').notNull().default(true),
  expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(), // 30 days from creation
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
});
```

**Connection setup:**

```typescript
// src/server/db/client.ts

import { drizzle } from 'drizzle-orm/neon-http';
```

## 3. File Storage

### Decision: No file storage. In-memory processing only. No Vercel Blob.

Screenshots are processed entirely in server memory:

```
Upload (client, compressed via Canvas API)
  -> FormData POST to /api/upload/screentime
  -> Buffer in memory (~500KB-3MB)
  -> base64 encode
  -> Anthropic Haiku 4.5 Vision API call (~2-4 sec)
  -> JSON response
  -> Buffer garbage collected
  -> Image never written to disk, database, or object storage
```

**Privacy guarantee (enforceable by architecture):**
- Vercel serverless functions have no persistent filesystem
- No `fs.writeFile`, no S3 SDK, no Blob SDK in the screenshot route
- The function dies after response; memory is released
- Anthropic API does not retain inputs by default for API customers

**Privacy policy language:** "Your screenshot is processed in server memory for approximately 3-5 seconds, then permanently discarded. It is never written to disk, database, or any storage system."

**Google Takeout (MVP, Weeks 7-10):**
- Parsed client-side using JSZip in the browser
- Only aggregated signals (category counts, time distributions) sent to server
- Raw ZIP never touches our servers
- No Vercel Blob needed

## 5. Rate Limiting & Cost Protection

### Decision: Upstash Redis (free tier) for 3 use cases

**Dependencies:**

```bash
pnpm add @upstash/redis @upstash/ratelimit
```

**Environment variables:**

```
UPSTASH_REDIS_REST_URL=https://xxx.upstash.io
UPSTASH_REDIS_REST_TOKEN=AXxx...
```

**Upstash free tier:** 10,000 commands/day, 256 MB storage. Sufficient for all 3 use cases at PoC/CV scale.

### Use Case 1: API Rate Limiting

Implemented as Next.js Edge Middleware (`src/middleware.ts`):

| Endpoint | Limit | Window | Rationale |
|----------|-------|--------|-----------|
| `POST /api/upload/screentime` | 5 requests | per IP per minute | Abuse prevention only. Screenshots are unlimited and free. |
| `POST /api/subscribe` | 3 requests | per IP per hour | Prevent email spam |
| `POST /api/billing/webhook` | No limit | -- | Stripe sends webhooks; never rate-limit Stripe |
| `POST /api/quiz/submit` | 10 requests | per IP per minute | Abuse prevention only. Quiz is free and unlimited. |

**Middleware implementation pattern:**

```typescript
// src/middleware.ts
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(5, '1 m'),  // 5 req/min — abuse prevention, not usage cap
  prefix: 'rl',
});
```

### Use Case 2: AI Cost Ceiling ($50/day)

Track Anthropic API spend per day. Implemented in the screenshot analysis route handler.

**Redis key pattern:** `ai_cost:YYYY-MM-DD` (TTL: 48 hours)

**Flow:**
1. Before Claude API call: `GET ai_cost:2026-03-30` -- check if under ceiling
2. If over ceiling: return 503 with "We're experiencing high demand. Try again tomorrow."
3. After Claude API call: calculate cost from `response.usage` (input_tokens, output_tokens)
4. `INCRBYFLOAT ai_cost:2026-03-30 <cost_in_cents>`

**At $0.003/screenshot, the $50/day ceiling allows ~16,600 screenshots/day.** At 100 users doing 3 screenshots each, daily spend is $0.90 -- 55x under budget. The ceiling is a safety net, not a constraint.

### Use Case 3: Session Counters (MVP)

Track per-user engagement for analytics and conversion optimization. E.g., count how many X-Rays a session has created to personalize CTAs ("You've created 3 X-Rays -- save them all with one click").

**Redis key pattern:** `usage:<session_id>:xrays` (TTL: 30 days)

## 9. Backup & Disaster Recovery

### What needs backing up:

| Data | Location | Strategy | RTO |
|------|----------|----------|-----|
| Source code | GitHub | Git history (inherent) | < 5 min (clone) |
| Database | Neon Postgres | Neon automatic PITR, 7-day retention (free) | < 30 min |
| Environment variables | Vercel Dashboard | Manual export quarterly to encrypted file | < 1 hour |
| DNS records | Domain registrar | Documented in private doc | < 30 min |

### What does NOT need backup:
- Uploaded screenshots (never stored)
- Build artifacts (regenerated on deploy)
- Vercel logs (transient)
- Node modules (regenerated from lockfile)

### Recovery procedures:

| Scenario | Action | Time |
|----------|--------|------|
| Bad deploy breaks production | Vercel Dashboard -> Deployments -> Instant Rollback | < 2 min |
| Database corruption | Neon Dashboard -> Restore to point in time | < 30 min |
| Vercel outage (prolonged) | Deploy to Netlify or Railway (standard Next.js) | < 4 hours |
| GitHub outage | Local git clones are full backups | Wait or push to secondary remote |
| Domain/DNS issue | Contact registrar; TTL is 300s for fast propagation | < 1 hour |
| API key compromised | Rotate immediately in Vercel Dashboard; redeploy | < 15 min |

```json
{
  "crons": [
    {
      "path": "/api/cron/purge",
      "schedule": "0 3 * * *"
    }
  ]
}
```

| Job | Schedule | Route | Purpose |
|-----|----------|-------|---------|
| Data purge | Daily 03:00 UTC | `/api/cron/purge` | Delete expired X-Ray results, enforce retention policy |
| Weekly playbook | Monday 08:00 UTC | `/api/cron/weekly-email` | Send weekly automation tip via Resend |
| AI cost report | Daily 00:05 UTC | `/api/cron/cost-report` | Email founder if daily AI spend > $25 |

Note: Vercel Cron on Hobby plan supports minimum 1-day intervals. Pro plan supports any cron expression. Data purge runs daily which works on Hobby. Weekly email works on Hobby. Cost report needs Pro (or run it as part of the daily purge job).

**Upstash QStash (event-driven jobs, MVP only):**

```bash
pnpm add @upstash/qstash
```

Used for operations that might exceed the 60s serverless timeout:
- Google Takeout server-side analysis (if client-side parsing sends aggregated data that needs Sonnet processing)
- Batch email sends (if Resend batch API is too slow for the function timeout)

**QStash free tier:** 500 messages/day, 50 messages/sec throughput. Sufficient for MVP.

**Why QStash over Inngest:** Same vendor as our Redis (Upstash). HTTP-based (no SDK, just POST to a URL). Simpler mental model. No step functions, but we don't need step functions at MVP scale. If we later need retryable multi-step workflows, evaluate Inngest then.

## 15. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Vercel Hobby hits serverless timeout (60s) on screenshot analysis | Low (Haiku is fast, ~2-4s) | High (blocked revenue) | Monitor p95 latency. If approaching 30s, optimize image size or upgrade to Pro. |
| Anthropic API outage during demo/launch | Low | High | Return cached/templated fallback response: "Analysis temporarily unavailable. Try again in a few minutes." No image is stored either way. |
| Stripe webhook endpoint fails silently | Medium | Critical (lost revenue) | Sentry alert on webhook errors. Stripe retries for 3 days. Idempotent handler with `stripe_session_id` unique constraint. |
| Neon free tier compute auto-suspends causing cold start delays | Medium | Low (adds ~1-2s on first query after idle) | Acceptable for PoC. Neon warms up within 1-2s. If problematic, keep-alive ping via Vercel Cron. |
| Upstash Redis free tier exhausted (10K cmd/day) | Low at PoC | Medium | Monitor daily command count. At 80% utilization, upgrade to pay-as-you-go (~$5/mo). |
| GDPR complaint before DPAs are signed | Low | High | Sign all DPAs in Week 1 before processing any user data. Checklist in Section 8. |
| Secret leak in git history | Low | Critical | `.env.local` in `.gitignore` (already true). No secrets in CLAUDE.md (current placeholders are fine). Add `gitleaks` to CI in MVP. |
| Bad deploy breaks Stripe checkout | Medium | Critical (lost revenue) | Vercel instant rollback (<2 min). Preview deploy testing before merge. CI build check catches type errors. |
