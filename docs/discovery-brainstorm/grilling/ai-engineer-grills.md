## 1. rapid-prototypes.md

### Hallucinations

**"Build time: 3 hours" for Subscription Autopsy.** This claim assumes: upload component already exists (it does, reuse ScreenTimeUpload), result card is trivial, and Claude Vision prompt works first try. The prompt engineering alone for subscription screenshots will take 2+ hours. iOS Subscriptions view has a very different layout from bank app transaction lists from Google Play's subscription view. You need at least 3 different prompt variants and test each against 10+ screenshots to get reliable extraction. Realistic build time: 6-8 hours for production-ready extraction across iOS + Android + bank screenshots.

**"Pure client-side. No AI needed" for Decision Fatigue Calculator.** True, but the document quietly inflates the numbers. "Weekly decision hours lost (typically 4-9 hours, which shocks people)" -- this range is generated by multiplying self-reported estimates by frequency. Self-reported time estimates are wildly inaccurate (research shows 2-3x overestimation). The tool will produce numbers that feel shocking but may not be trustworthy. This isn't a hallucination per se, but it's building a viral tool on shaky data. Fine for entertainment, risky if users take the numbers seriously.

**"Context switching back from email costs 23 minutes each time (UC Irvine research)" in Email Avalanche Audit.** This is a misuse of the Gloria Mark research. The study found that it takes ~23 minutes to return to a task after an interruption, NOT that every email check costs 23 minutes. An email check IS the task if you're doing email. The 23-minute cost applies when you're interrupted FROM a non-email task. Multiplying 34 checks x 23 minutes = "13 hours of lost deep work per week" is mathematically absurd -- there are only ~40 working hours in a week. This would mean email alone consumes 33% of a work week in context-switching, on top of the actual email time. The real number is significant but nowhere near 13 hours. Don't cite this research this way -- it's the kind of claim that gets fact-checked on Twitter and makes the whole product look unserious.

### Feasibility Gaps

**Copy-Paste Detective (#10).** "Paste any block of text from your clipboard -- we'll tell you what you were trying to do." This sounds clever but the value proposition is extremely thin. Someone pastes an address. We say "you're probably sending this to someone." No kidding. The insight is obvious in 80% of cases. The 20% where it's genuinely useful (tracking numbers, repeated email templates) requires the user to paste multiple clipboard items over time to detect patterns -- a single paste tells you almost nothing about automation opportunities. This feature would feel gimmicky after one use.

**Shopping Spiral Tracker (#7) using browser tab screenshots.** "Extract tab titles, identify shopping sites." Claude Vision can extract tab titles from the Chrome tab grid view on mobile, but the tab THUMBNAILS are tiny and low-resolution. On desktop, the tab bar shows truncated titles (usually just the first 15-20 characters). The claim that we can identify "19 of them are shopping" from tab titles is optimistic. Many shopping tabs have generic titles like "Search Results" or a product name that doesn't obviously signal shopping. You'd need full URLs, not just titles, and you don't get URLs from a screenshot.

### Cost Underestimates

The build time estimates are universally optimistic by 2-3x. "3 hours" for a feature means: no error handling, no edge cases, no mobile responsive testing, no loading states, no empty states, no analytics tracking. For the Vision-based features (Subscription Autopsy, Notification Flood Score, Doomscroll Meter, Shopping Spiral), add 1-2 hours just for prompt engineering and testing across different screenshot formats.

Total claimed: "6 new discovery angles in 3 days" (13 hours total). Realistic: 5-7 days of actual development including testing and edge cases.

### Privacy Risks

None significant. The document correctly sticks to screenshots the user takes voluntarily. The bank statement screenshot (in Subscription Autopsy) is the riskiest source but is explicitly listed as optional. No red flags.

### What's Actually Brilliant

- **The tiered approach: questionnaire-only features first, then Vision-based.** Building Decision Fatigue Calculator and Sleep Procrastination Score as pure client-side tools with zero API cost is the right move. They validate the "discovery tool" pattern before you spend money on Vision API calls.
- **Revenue connection framing.** Every feature ties directly to a paid tier. This isn't just insight -- it's a sales funnel. Well done.
- **Content Creation vs Consumption Ratio.** The 68:1 ratio IS meme-ready. This feature will get screenshotted and shared. It's the highest-ROI feature in the list relative to build effort.
- **The "gravity app" concept in Doomscroll Meter.** Naming the most-opened-first app as the "gravity app" is excellent product language. Sticky, shareable, slightly accusatory.

## 2. discovery-engine.md

### Hallucinations

**"$0.002 per parse" for Screen Time OCR.** Let's check the math. Claude Haiku Vision pricing (as of early 2026): input tokens include the image (~1600 tokens for a typical screenshot at recommended resolution) plus the prompt (~200 tokens), output ~300 tokens. At $0.25/MTok input and $1.25/MTok output for Haiku, that's roughly (1800 * 0.25/1M) + (300 * 1.25/1M) = $0.000825. So ~$0.001 per parse, not $0.002. The $0.002 figure is conservative (safe side), so this isn't a problem -- but it does mean the "$0.003/typical user" budget is actually more like $0.002. The cost story is BETTER than claimed, not worse.

**"Pattern narrative generation: Claude Haiku ($0.001/report)."** This is plausible for a simple template-fill but unrealistic for the kind of multi-paragraph, personalized narratives described in the examples. Those examples read like Sonnet-quality output. Haiku can do a decent narrative, but the nuanced, witty, slightly confrontational tone shown in the examples ("You're not procrastinating sleep. You're reclaiming 'me time' your day didn't give you") requires Sonnet. Budget $0.005-0.01 for the narrative step, not $0.001.

**"Cross-source fusion examples" -- the Browser + Screen Time "intent vs. action" detection.** The example shows: "45 minutes browsing recipe sites" followed by "Uber Eats app opened within 30 min of recipe browsing." This requires correlating browser history timestamps with Screen Time timestamps at the minute level. Screen Time screenshots give you daily/weekly aggregates, NOT per-session timestamps. You cannot determine that Uber Eats was opened "within 30 minutes of recipe browsing" from a Screen Time screenshot. This level of correlation requires either the Chrome extension (live tab tracking) or Google Takeout (timestamped Chrome history) combined with the macOS knowledgeC.db (timestamped app usage). Two screenshots cannot produce this insight. The document presents it as achievable at Level 2-3 of the discovery funnel, but it actually requires Level 5 data (Takeout or extension).

**"Each notification costs ~23 seconds of context-switch time" in the Interruption Cost Analysis.** Same misuse of the Gloria Mark research as in rapid-prototypes.md. The 23-minute figure is about returning to a primary task after an interruption, and the document downgrades it to 23 seconds per notification without citing any source for that number. Where does 23 seconds come from? It seems made up to produce a nice calculation (178 x 23s = 68 min). If you're going to cite context-switching research, use real numbers or be transparent that the estimate is rough.

### Feasibility Gaps

**The "Meal Planning Fail" detector.** Requires detecting that a user browsed recipe sites AND opened a delivery app within a time window. As noted above, this requires timestamped cross-source data, not screenshots. The screenshot-based discovery funnel cannot detect temporal sequences between apps. This pattern detector should be moved to the "requires Chrome extension or Takeout" tier, not presented as achievable with the screenshot pipeline.

**The "Network Effect Moat."** "The 1,000th user gets a better experience than the 1st user because the pattern library and suggestion templates have been refined by aggregate behavior." This is a real feedback loop, but calling it a "network effect" is a stretch. A network effect means the product gets more valuable as more users join because users interact with each other. This is a data flywheel (more data = better model), which is a weaker moat. Competitors can replicate a data flywheel by buying training data. Be precise about the moat type.

**Rule-based pattern detectors "50-100 lines of TypeScript each."** The simple ones (zombie subscription, notification flood) are indeed 50 lines. But "Research-Then-Buy" and "Context Switching" require temporal correlation across sources with windowed joins. That's closer to 200-400 lines with edge cases, timezone handling, and confidence scoring. Don't underestimate the complexity of temporal pattern matching.

### Cost Underestimates

**"2-3 days of development" for 10-15 pattern detectors.** No. The simple threshold-based detectors (zombie subscription, notification flood, email overload) are 2-3 hours each. The temporal correlation detectors (context switching, research-then-buy, meal planning fail) are 1-2 days each because they require building the temporal join infrastructure first. Realistic total: 1-2 weeks for 10-15 detectors including the temporal engine.

### Privacy Risks

**"System prompt: Here are the user's detected patterns: {patterns_json}."** The document says "NEVER send raw data. Only send aggregated patterns." This is good in principle, but the pattern summaries as described (e.g., "user visits Zillow 47 times/day", "Gmail usage 47 min/day with 120 emails") contain personally identifiable behavioral data. Under GDPR, this is personal data and must be treated as such. The distinction between "raw data" and "patterns" doesn't exist in GDPR -- if the pattern identifies a specific user's behavior, it's personal data. The privacy story should focus on: (1) data processed client-side where possible, (2) data retention limits, (3) right to deletion, not on the false dichotomy of "raw vs. aggregated."

### What's Actually Brilliant

- **The insight quality multiplier curve.** The idea that each new data source doesn't just add linearly but multiplies the value of all previous sources is the correct mental model and the correct product architecture. This is the real moat.
- **The two-tier suggestion engine (rule-based + AI narrative).** Keeping 90% of the logic as deterministic rules and using AI only for narrative generation is the right cost architecture. It also makes the system testable and debuggable.
- **"Why Apple/Google/ChatGPT Cannot Replicate This" section.** The conflict-of-interest argument is genuinely strong. Google will never tell you to use YouTube less. Apple will never cross-reference Health with Wallet. This is Meldar's structural advantage.
- **Cost circuit breakers.** Per-user limits, daily spend caps, founder alerts at 80%. This is production thinking, not brainstorm fluff.

## 3. architecture.md

### Hallucinations

None significant. This is the most grounded document of the six. The architecture is pragmatic: JSONB signals, computed profiles, pluggable processors, additive migrations. No wild claims.

### Feasibility Gaps

**"Adding a new source is one file, one registration call, zero new routes, zero new tables."** Almost true. But each new source also needs: (1) a Vision prompt or parser (the hard part), (2) a signal mapper with domain-specific logic (categorization, deduplication), (3) frontend upload component modifications (different upload hints, validation, result rendering). The "one file" claim covers the backend plumbing but ignores the prompt engineering and frontend work. Each new source is realistically a 1-2 day effort, not a one-file change.

**Side-effect import registration pattern.** `register()` called from a side-effect import in `src/server/discovery/sources/index.ts`. This pattern is fragile in Next.js because of tree-shaking and module bundling. If the bundler decides the import has no visible exports or side effects, it may be eliminated. This works in Node.js but may not work in Next.js Edge Runtime or with Turbopack's tree-shaking. Use an explicit registry initialization function called from the route handler instead of relying on side-effect imports.

**The `raw_input JSONB` column stores the AI extraction verbatim.** If this includes the original screenshot (base64), rows could be 500KB-2MB each. If it's just the extracted structured data, it's fine (a few KB). The document doesn't clarify. Storing base64 screenshots in JSONB would be a database anti-pattern -- use object storage (S3/R2) for images.

### Cost Underestimates

**Migration and schema work: "Phase 1: Add new tables alongside."** The migration is clean, but the refactor of `insights.ts`, `upsells.ts`, `suggestions.ts` to consume `Signal[]` instead of `ScreenTimeExtraction` is more work than implied. Those files likely have hardcoded assumptions about the shape of Screen Time data. Refactoring them to be source-agnostic while maintaining backward compatibility with the existing `xray_results` path is a 1-2 day effort, not a quick wrap.

### Privacy Risks

**`raw_input JSONB` stores extracted data with no explicit retention policy.** The document mentions no TTL, no auto-deletion, no data retention limits. Under GDPR, you need a legal basis for storing personal data and a defined retention period. Add: (1) auto-delete sessions after 90 days of inactivity, (2) explicit `DELETE /api/session/:id` endpoint for right-to-erasure, (3) don't store raw screenshots in the database at all -- process and discard.

### What's Actually Brilliant

- **Computed profiles, not materialized.** At PoC scale, this is exactly right. No cache invalidation. No stale data. Just re-compute from source signals on every request. Elegant.
- **Deduplication strategy: "take the higher usage value."** This is the right heuristic for screen time data. Users upload multiple screenshots hoping for accuracy. Taking the max avoids underreporting while being simple to implement.
- **Phase-based migration.** Keeping `xray_results` alive while building the new system alongside it is the safe approach. Zero risk to existing data. Ship the new system, verify it works, migrate later.
- **The processing pipeline abstraction.** `validate -> extract -> normalize -> persist -> aggregate` as a fixed pipeline with pluggable implementations per source is clean, testable, and extensible. This is good software architecture.

## 4. email-discovery.md

### Hallucinations

**"232 total notifications/day" and "Telegram 178/day" presented as representative examples.** These are extreme outlier numbers. The average smartphone user gets 46-80 notifications per day (various studies, 2023-2025). 232/day is 2-3x the norm. Using outlier examples makes the product demo look impressive but sets unrealistic expectations. Most users will see 50-100 notifications, and the insight will be "you get 67 notifications a day" -- which is less shocking. Use realistic numbers in examples or explicitly frame the 232 case as an extreme.

**"Each notification costs 23 seconds of context-switch time."** Third time this number appears across documents. Still no source. Still not the Gloria Mark number (23 minutes, not seconds). If this is a Meldar-specific estimate, say so. If it's from research, cite it. Presenting an unsourced number as fact across three documents is a pattern that needs fixing. Pick a defensible number with a real citation or explicitly label it as an estimate.

**"You're in 23 Telegram groups but active in 3."** The document claims to detect this from a notification center screenshot + Screen Time data. The notification screenshot shows notification COUNT from Telegram, not the number of groups. Screen Time shows total Telegram time. Neither tells you how many groups the user is in or which ones they're active in. You'd need the user to screenshot their Telegram chat list AND count groups. The "23 groups, active in 3" insight requires data you don't have from the proposed inputs.

### Feasibility Gaps

**"Unified Inbox View" for messaging apps.** "One dashboard showing unread counts across all messaging apps (via screenshot upload or Screen Time data)." Screen Time doesn't show per-app unread counts. Screenshots of individual messaging apps show unread counts, but building a "unified dashboard" from screenshots requires the user to upload a new screenshot every time they want to see the dashboard. This isn't a dashboard -- it's a one-time snapshot. Don't call it a view or dashboard.

**Email "receipt" search screenshot approach.** Asking users to search their inbox for "receipt" and screenshot the results is clever, but the screenshot will show maybe 10-15 email rows. The user may have 200+ receipt emails. They'd need to scroll and take 15+ screenshots. This doesn't scale. The Gmail Takeout export or MBOX upload is the right approach for comprehensive email analysis, not screenshots of search results.

### Cost Underestimates

None significant. This document is mostly about extraction techniques, not build estimates.

### Privacy Risks

**Bank statement screenshots.** The document correctly identifies this as sensitive but doesn't address: what if the screenshot captures account numbers, balance, or transaction details beyond subscriptions? Claude Vision will extract everything visible. If the prompt says "extract recurring charges" but the screenshot shows a $3,000 paycheck deposit and an overdraft fee, the model will see that data even if we don't store it. The image IS sent to Claude's API. Anthropic's data retention policies apply to that API call. Users need to understand that their bank screenshot is processed by a third-party AI. This should be disclosed.

**The notification preview text extraction.** "WhatsApp messages, email subjects, private DMs" visible in notification screenshots. The document acknowledges this but the mitigation is weak: "we should warn users." A warning isn't enough for GDPR. If you're extracting notification preview text, you're processing communication content. Under GDPR Article 9, this could touch special categories of data (health messages, political opinions in group chats). The safer approach: prompt the Vision model to extract ONLY app names and notification counts, explicitly instructing it to ignore preview text content. You can do this reliably with Claude -- add "Do NOT extract or return any message content, preview text, or message body. Only extract app names and notification counts" to the prompt.

### What's Actually Brilliant

- **The notification-to-usage ratio.** This is genuinely a novel metric that no existing product surfaces. The ratio between "how many times an app interrupts you" and "how much time you actually spend in it" is powerful and actionable. This is the kind of insight that makes someone mute a group chat on the spot.
- **The "35 seconds, 4 screenshots" pitch.** Compelling. The total-time-to-insight framing is the right way to pitch screenshot-based discovery. Make this the headline, not the technical details.
- **Communication pattern taxonomy.** "Passive Group Chat Member," "Platform Fragmentation," "Notification Anxiety Loop" -- these named patterns are excellent product language. People will identify with them. "I'm a Platform Fragmenter" is a shareable self-diagnosis.
- **Actionable fixes table.** Every pattern maps to a specific, concrete fix with implementation notes. "Mute the 15 least-active groups in 2 minutes" is actionable. "Consider reducing your notifications" is not. This document gets the action layer right.

## Cross-Document Issues

### The 23-second/23-minute context-switching problem

This number appears in three documents (rapid-prototypes, discovery-engine, email-discovery) with inconsistent usage. The original research (Gloria Mark, UC Irvine, 2004) found it takes ~23 minutes to return to a task after interruption. The documents variously cite this as "23 minutes per email check" (wrong -- email IS the task), "23 seconds per notification" (unsourced), and "68 minutes of invisible lost focus" (derived from the unsourced 23-second figure). Pick ONE defensible number, cite it properly, and use it consistently. Recommended: "Research suggests it takes 23 minutes to refocus after an interruption (UC Irvine). Even brief phone checks disrupt concentration." Don't multiply it into an inflated total.

### Haiku vs. Sonnet for narratives

Multiple documents budget Haiku ($0.001) for generating the Time X-Ray narrative. But the example outputs across all documents show witty, psychologically insightful, multi-paragraph narratives with personality and confrontational tone. That's Sonnet output, not Haiku. Haiku generates competent but bland text. If you want "You're not procrastinating sleep. You're reclaiming 'me time' your day didn't give you" -- that's Sonnet. Budget accordingly: $0.005-0.01 per narrative, not $0.001.

### The temporal correlation blind spot

Three documents (discovery-engine, email-discovery, SYNTHESIS) propose cross-source insights that require minute-level temporal correlation between app usage sessions (e.g., "opened Uber Eats within 30 minutes of recipe browsing"). This is NOT achievable from the screenshot pipeline. Screenshots give you daily/weekly aggregates, not per-session timestamps. These insights require either:
1. macOS knowledgeC.db (per-session timestamps, macOS only)
2. Chrome extension (live tab tracking + idle detection)
3. Google Takeout Chrome history (timestamped URLs)

None of these are available in the "30-second screenshot" onboarding flow. These insights should be explicitly labeled as "Level 4-5 only" and not presented alongside screenshot-based insights as if they're comparable in ease.

### Missing: Claude Vision accuracy claims need testing

Multiple documents claim "90-95%+ accuracy" or "near-perfect accuracy" for Claude Vision extraction from screenshots. These claims are plausible for English-language, light-mode, recent iOS versions. But they haven't been tested against:
- Dark mode (inverted colors, different text rendering)
- Non-English languages (especially RTL languages, CJK characters)
- Android manufacturer skins (Samsung One UI, Xiaomi MIUI, OnePlus OxygenOS -- all look different)
- Older iOS versions (iOS 15/16 have different Screen Time layouts)
- Low-resolution screenshots (some phones capture at lower DPI)
- Screenshots with accessibility settings (large text, bold text, reduced motion)

Before shipping, run the Vision prompts against a test suite of 50+ screenshots across these variants. The "95% accuracy" claim is probably true for iPhone 15 + English + light mode. It might be 70% for Samsung Galaxy + Turkish + dark mode. Know your failure modes before users hit them.

### Missing: What happens when extraction fails?

None of the six documents address the failure mode: what if Claude Vision can't extract the data from a screenshot? User uploads a blurry photo, a screenshot of the wrong screen, or a screenshot in a language the model handles poorly. The user experience at failure is critical:
- Don't show an error page. Show a partial result with a "we couldn't read some of your data" message.
- Offer manual entry as a fallback: "We couldn't read your screen time data. Enter your top 5 apps manually."
- Log the failed screenshot (with user consent) to improve prompts.

This is a UX and engineering concern that needs to be addressed in the architecture.

## Bottom Line

**Strongest document:** architecture.md. Pragmatic, grounded, no wild claims. Ship it as-is with the side-effect import fix.

**Most valuable insight across all docs:** The cross-source insight matrix (discovery-engine.md) and the revenue path table (SYNTHESIS.md). These define what Meldar IS.

**Biggest risk across all docs:** The temporal correlation blind spot. Half the "killer insights" require data that the screenshot pipeline can't provide. Be honest about what Level 1-2 can actually deliver and don't overpromise in the UI. The screenshot-only insights (notification ratio, zombie subscriptions, inbox overwhelm) are genuinely powerful on their own. Don't dilute them by promising temporal insights you can't yet deliver.

**Biggest missing piece:** ChatGPT conversation export isn't in the SYNTHESIS.md priorities. It should be #2 after Screen Time for the Gen Z audience.

**Fix immediately:** The 23-second context-switching number. It appears in three documents, is unsourced, and will embarrass you if a journalist or skeptical user fact-checks it.
